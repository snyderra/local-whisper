## 1. Architecture decisions and why

**Process topology is inverted for app installs.** Historically the LaunchAgent ran the Python
service (`wh _run`), which spawned the Swift UI from `~/.whisper/LocalWhisperUI.app`
(`app.py:_spawn_swift_ui`). For `INSTALL_APP` the LaunchAgent runs
`Local Whisper.app/Contents/MacOS/LocalWhisperUI`, and `ServiceManager.swift` spawns the bundled
Python service as its child. Reason: launchd's TCC "responsible process" is the job executable.
With the UI as parent, Microphone/Accessibility grants attribute to the signed app bundle (stable
name, icon, and — once Developer ID exists — identity across updates). A launchd-run bare
`python3.12` would get its own unstable identity, forfeiting Tier 2's core payoff. Dev installs are
untouched: the inversion only activates when Info.plist has `LWBundledRuntime` (dmg builds).

**Supervision contract (two supervisors that must not fight):** launchd keeps the *UI* alive with
`KeepAlive={SuccessfulExit=false}`; `ServiceManager` mirrors exactly that for the python child —
restart on crash with a 10s→120s backoff, **never restart on exit 0**. The service deliberately
exits 0 when mic permission is missing and when another instance holds the single-instance lock;
both cases must not hot-loop. `wh stop` does `launchctl bootout` *before* pid-killing so launchd
doesn't resurrect the UI (killing the UI by signal counts as unsuccessful exit).

**Single-instance + stale-version reconciliation:** `FirstRunManager.ensurePrimaryInstance()` —
if another instance of the same bundle id runs, activate it and `exit(0)` (which launchd treats as
"don't restart"). Before exiting, `reconcileStaleService()` compares this bundle's version against
the running service's (via hidden `wh _service_version`, which reads the `version` field added to
the cmd-socket `status` payload in `app_commands.py::_cmd_status`); mismatch →
`launchctl kickstart -k gui/$UID/com.local-whisper` restarts the whole job from disk. This heals
manual Finder drag-upgrades, not just Sparkle updates. Note: when the service is down,
`wh _service_version` prints nothing on stdout (client error goes to stderr) — the Swift probe
treats empty as "no reconcile".

**LaunchAgent ownership:** the plist (same label/path as always: `com.local-whisper` at
`~/Library/LaunchAgents/`) is generated by `src/whisper_voice/_launchagent.py` (plistlib, pure
`render_plist()` for tests) and exposed as hidden `wh _agent install|uninstall|status`. The Swift
`FirstRunManager` shells out to it (single source of truth shared with `wh setup` and
`wh doctor --fix`) but reads the plist directly for the staleness check. **No `EnvironmentVariables`
in the plist** — the Swift parent sets the child env explicitly (`PATH` with bundle bin first,
`HF_HUB_CACHE`, `LOCAL_WHISPER_UI_PARENT=1`), with `_ensure_runtime_env()` in `service_main` as
defense-in-depth (it also re-links `~/.whisper/bin/ffmpeg` into the current bundle after updates).
Translocated bundles (`/AppTranslocation/` in the path) never get written into the plist — alert
the user to move to /Applications instead.

**`INSTALL_APP` detection** (`_install.py`): positional scan of resolved `sys.prefix` for any
`*.app/Contents/Resources` component — survives renames, any install dir, and Gatekeeper
translocation. Checked before brew/source/pip. `get_app_bundle_root()` is the shared accessor
(both lru_cached — tests must `cache_clear()` both).

**Python embedding:** uv-managed python-build-standalone 3.12 copied into
`Contents/Resources/python`, project pip-installed **non-editably** into its own site-packages.
**No venv** — `pyvenv.cfg` bakes absolute paths and would put `sys.prefix` outside the bundle,
breaking both relocation and detection. Launcher shims in `Contents/Resources/bin`: `wh` (sh) →
`python3.12 -s -E -B wh.py`; `wh.py` is a real script (not `-m whisper_voice`) so
`os.execv(sys.executable, [sys.executable] + sys.argv)` restarts keep working. `-B` everywhere:
a `.pyc` write inside Resources would invalidate the code-signing seal (verified: seal intact
after a service run).

**Versioning:** bundle Info.plist version comes from `pyproject.toml` at build time (single
source). The `setup.sh`/`cli/build.py` hardcoded duplicates still serve the legacy dev bundle and
are bumped by `release.sh` as before. The service now reports its version over the cmd socket.

## 2. Hard-won gotchas (each cost real debugging)

1. **PEP 668:** uv-managed PBS trees ship `lib/python3.12/EXTERNALLY-MANAGED`; pip refuses to
   install. The copy is ours, so `build_bundle.sh` deletes the marker from the copied tree.
2. **Universal2 otool parsing:** `otool -L` on fat binaries prints a *second unindented header
   line per architecture* which naive parsers read as a dependency (→ 230 false audit failures).
   Only tab-indented lines are load entries.
3. **Stale wheel rpaths:** scikit-learn wheels carry `LC_RPATH /Users/runner/miniconda3/envs/build/lib`
   from their build machines — dead in every pip install, zero `@rpath` loads actually use it.
   `build_bundle.sh` strips absolute rpaths with `install_name_tool -delete_rpath` **and must
   ad-hoc re-sign each modified file** (arm64 refuses to load invalidated signatures).
4. **Dangling symlinks break codesign:** deleting pip's absolute-shebang scripts from
   `python/bin` orphaned `2to3`/`idle3`/`pydoc3`/`python3-config` symlinks; `codesign --verify
   --deep --strict` then fails with a misleading "No such file or directory" on the app itself.
   The build now deletes dangling links.
5. **The process-matcher shell-kill:** the service's duplicate sweep SIGTERMs anything matching
   the service predicate. Substring-anywhere matching meant *a shell command that merely
   mentioned* `Local Whisper.app` and `wh.py _run` (e.g. in a grep pattern) got killed — it
   killed the implementing session's own shell twice. `_service_identity.is_service_command`
   is now positional: command must **end** with the `_run` token, identity markers must be in
   the head. Also fixes the original bug: `local-whisper` (hyphen) never matched
   `Local Whisper.app` (space) paths, so `wh stop` couldn't find bundle services at all.
   Regression tests included.
6. **`find` + `pipefail`:** `find nonexistent-dir | while … | xargs` aborts the script *after*
   the pipeline completes (signing worked, then died before the outer-app step, leaving a
   half-signed bundle that looked linker-signed). Guard optional dirs before putting them in
   pipelines.
7. **Sign inside-out, never `--deep`:** leaves (all nested Mach-Os by 4-byte magic — wheels ship
   extensionless binaries) → frameworks (inner XPC first) → `python3.12` + `ffmpeg` **with
   entitlements** (entitlements bind per-executable and the mic-using process IS python) → outer
   app. Ad-hoc needs `--timestamp=none`.
8. **ffmpeg must be a copy, not a symlink** into site-packages — codesign needs a real file.
9. **uv python selection:** `uv python find 3.12` happily returns MacPorts' interpreter;
   `UV_PYTHON_PREFERENCE=only-managed` forces the relocatable PBS build. The uv dir name may be
   a symlink — resolve with `readlink -f` before `cp -R` (BSD cp copies the link itself).
10. **TCC responsible-process facts** (validated across Tier 1 + 2): CLI-session processes
    attribute to the terminal app (here: Supacode); launchd-job processes attribute to the job
    executable; children inherit the parent app's responsibility. This drove both the Tier 1
    "grants show as Python" messaging and the Tier 2 inversion. Also: `tccutil` cannot remove
    path-identified grants — uninstall flows can only point users at the Settings pane
    (stale grants are inert once the binary is deleted).
11. **Mic-denied startup:** `service_main` retries mic 3×(3/6/9s) then **exits 0** — by design,
    so neither launchd nor ServiceManager hot-loops. A "Cannot connect to service" right after
    start on an ungranted machine is this, not a bug.
12. **pip resolver backtrack:** `numpy>=2.4.4` can drive resolvers to numba 0.53 (unbuildable
    llvmlite). Bundle build falls back to `uv pip install … "numba>=0.60"`; the dev test venv
    needs the same pin.

## 6. Release pipeline shape

`scripts/release.sh X.Y.Z` (unchanged flow: bump → CHANGELOG → test → tag → `gh release create`
→ brew formula bump) → the **published release** triggers `release-app.yml`: Xcode 26 required
(hard fail), tag==pyproject guard, signing mode resolved from `MACOS_CERT_P12` presence (absent →
ad-hoc `-adhoc.dmg`, uploaded for smoke but never appcast-referenced), build → sign → notarize
(self-skips) → dmg (>1.8 GB hard fail) → `gh release upload` → with Developer ID +
`SPARKLE_ED_PRIVATE_KEY`: Sparkle `sign_update` (pinned 2.8.0 tools tarball), `make_appcast.py`
prepends the item, commit `appcast/appcast.xml` to main, `gh workflow run docs-pages.yml`
(GITHUB_TOKEN pushes don't trigger workflows). Appcast serves from the existing Pages site at
`/appcast.xml`. Known risk: the appcast push may hit branch protection → fallback is a PAT secret
or switching `SUFeedURL` to the `releases/latest/download` pattern.
